第一更：轻装上阵，梦想起航

2019.03.17

偶然间看到了 Lusifer 课堂视频，获益匪浅，见识了资深程序员的开发思想，并定制自己的学习路线。

有道无术，术尚可求；有术无道，止于术。

———— 《道德经》

2018年还很少听到企业采用微服务框架，如今的企业开发，微服务已大行其道。

我认为，当下的 Java 程序员，入门的标准，不再是曾经的 SSM，起码有微服务的编程思想。
思考问题要有一定的高度，当拥有降维打击的能力时，那就无坚不摧了。

微服务这个概念是2012年出现的，作为加快 Web 和移动应用程序开发进程的一种方法。2014年开始受到各方的关注，同年为微服务的元年。

2019重点学习计划一：微服务

为何抛弃单体应用

  1. 单体应用随着时间的推移会越来越臃肿，十分不利于分析代码的结构和逻辑关系
  2. 庞大的项目对于开发新功能和修复 Bug 十分耗时（焦油坑）
  3. 复杂的单体应用本身就是持续部署的障碍，启动时间过长并且容易影响其他模块
  4. 不同模块存在资源需求冲突时，单体应用难以扩展
  5. 单体应用所有应用处于同一进程中，任何模块的问题都将拖垮所有模块
  6. 单体应用使得采用新框架和语言将非常困难

何为微服务（撸帝独创水疗大法）
微服务架构好比一家水疗会所，
技师是服务提供者，你是服务消费者，水疗会所是注册中心。
你去水疗会所找技师，技师来提供各种服务。
一个技师如果不在还能换下一个技师，这就是高可用。
只要记住了号牌，下次还能叫这个技师，这就是缓存。
如果技师离职了，你所记住的号牌也就作废，这就是缓存穿透。
后来你没有号牌了，经理带你安排技师，这就是缓存击穿。
同一时间你有认识的技师都离职了，这就是缓存雪崩。
因为要额外安排技师，所以速度会慢一些。

精髓：熔断机制（服务挂了之后返回固定的结果）
水疗会所只有足疗，谁问都一样，因为，警察来了。

如今常用的微服务框架解决方案有 Spring Cloud 和 Dubbo，出自 Spring 和阿里之手。
不过广义的来讲它俩一个是编程模型，一个是RPC通信框架，没什么相通点。
技术没有好坏，只有是否合适。
应该根据业务来组合式的使用各种框架，没有必要去钻牛角尖的分析他们的利弊优劣。
比如需要做的一个对内RPC，对外REST的开发框架：Spring Cloud Alibaba Dubbo

所有微服务的开发，都在思考四个问题：

  1. 多服务之间客户端如何访问
  2. 多服务之间服务端如何通信
  3. 多服务之间如何管理
  4. 多服务之间如何解决服务宕机

其实，如今的微服务还没有全球统一的标准，不过很有可能是未来的ServiceMesh（服务网格化）。

程序员也好，一家互联网企业也好，他应有四个阶段的发展过程：

  1. 入门期：主要就是微服务架构的基础建设来实现项目交付
  2. 发展期：以大数据为依据进行精准营销
  3. 成熟期：大数据+人工智能的部分实现
  4. 未来规划：人工智能+区块链，全球计算机共同实现的超级计算机

我现在处于第一阶段，准备向第二阶段进发。
如今BAT等已经在第四阶段深耕多时，然而很多企业还没有步入第一阶段，大厂不愧为大厂啊。

第二更：技术变革，思想先行

2019.07.16

记录最近学习微服务的一些心得。

天地无极，万法归一：康威定理

优点

  • 解决了复杂问题，在原系统功能数量不变的情况下，使个体服务更快的被开发，并更容易理解和维护
  • 每个服务的独立开发，使得当前技术的重写旧服务变得可行
  • 微服务架构模式可以实现每个服务独立部署，使持续部署成为可能
  • 微服务架构模式使得每个服务可以独立扩展，可以仅部署每个满足服务的容量和可用性约束的实例条目

缺点

  • 目前微服务边界不够明确
  • 分布式系统使得整体变得复杂，模块间的互相调用都需要做相应的故障处理
  • 分区数据库架构是一项挑战，不仅是CAP定理的限制，并且不支持如今高度扩展的NoSQL数据库和消息代理（分布式事务）
  • 测试、配置、部署、扩展、监控都会很繁琐

CAP 定理与 BASE 理论

CAP 定理

一个分布式系统，最多只能同时满足一致性、可用性和分区容错性这三项中的两项。

  • 一致性（Consistency）:更新操作成功并返回客户端后，所有节点在同一时间的数据完全一致
  • 可用性（Availability）:服务在正常响应时间内一直可用
  • 分区容错性（Partition tolerance）:分布式系统在遇到某节点或网络分区故障时仍能对外提供满足一致性和可用性的服务

CAP 权衡

CP系统：金融级系统
AP系统：和钱无关的系统

BASE 理论

即使无法做到强一致性（Strong Consistency），但应用可以采用适合方式达到最终一致性（Eventual Consistency）

  • 基本可用（Basically Available）：分布式系统出现故障时允许损失部分可用性，即保证核心业务可用
  • 软状态（Soft State）：允许系统存在中间状态，该中间状态不会影响系统整体可用性。分布式存储一般一份数据至少会有三个副本，允许不同副本同步的延时就是软状态的体现（同比MySQL中 Replication 的异步复制）
  • 最终一致性（Eventual Consistency）：系统中所有数据副本经过一段时间后最终达到一致状态。弱一致性和强一致性相反，最终一致性是弱一致性的一种特殊情况

ACID 和 BASE 的区别和联系

ACID 是传统数据库常用的设计理念，追求强一致性模型。
BASE 支持的是大型分布式系统，提出牺牲强一致性获得高可用性。
在分布式系统中，系统组件对于一致性的要求是不同的，ACID 和 BASE 会结合使用。

如何应对高并发

定义

高并发（High Concurrency）是互联网分布式系统架构设计中必须考虑的因素之一。通常是指：通过设计保证系统能够同时并行处理很多请求。

  • 响应时间（Response Time）：系统对请求作出响应的时间。
  • 吞吐量（Throughput）：单位时间内处理请求的数量
  • QPS（Query Per Second）：每秒响应请求数
  • 并发用户数：同时承载正常使用系统功能的用户数量

提升策略

垂直扩展
  • 增强单机硬件性能。如CPU增核，硬盘扩容等
  • 提升单机架构性能。如使用 cache 减少 IO 次数，使用异步增强吞吐量，使用无锁数据结构来减少响应时间等
水平扩展（主要方法）

  • 服务层：多机器负载均衡
  • 数据库：分库、分表、表分区

第三更：蓦然回首，寂寂寥寥

2019.09.27

正值青年，回首再读鲁迅的《从百草园到三味书屋》和朱自清的《背影》，又多了不一样的感触。
其实我想说的是，中学的少年时代，读这些触目感怀的文章，是很难理解作者的心情的。
唯有离家远去，或求学，或工作，那些与家人长期分别的日子里，才能隐约体会到作者那惆怅的心境。
古往今来，从不缺悲戚感怀的诗人，他们应该都大抵如此罢。

不，那些优秀的散文、杂记，是可以让每个人都读到不一样的东西的。
少年有少年的烦恼，青年有青年的惆怅，到了中年和老年，再翻出这经典的文章，
怕是又能增加不少与作者共鸣的感伤了。

辗转六年，从求学，到工作，一路奔波。
有看文章说，三十岁前，莫回头，不要怕，尽管先前冲，去试错，去见天地；
三十岁后，要自省，要谨慎，去见自己。
按说不应该在这个朝气蓬勃的年纪去庸人自扰，但是闲来时总想无病呻吟几句。

前几天跟母亲打电话，得知母上已经五十有余，顿时感到时间确实过得快了，
不由想到了子欲养而亲不待，一阵后怕。
父亲的身体早已远不如前，去年的一场急病也甚是后怕，
和他们相处的时间，可能远比我想的要少了。
今年国庆节，本不打算回去的，可是念及于此，还是买了回家的火车票，想着能双方都有时间的假期真是不多了。
本打算假期一起驱车去登泰山，趁他们现在脚力尚好。
可是看到往年国庆假期泰山的人流和六小时的舟车颠簸，想想还是搁置了。

今年姐姐终于也是顺利毕业并有了一份稳定的工作，十分欣喜。
昨日在与姐姐的通话中，得知她的我的关心甚至超过对自己，十分感动。
不过姐姐由于多年的生活习惯和刚工作的缘故，和我有一些不同：
她过于看重钱的数量，忽略了钱之外的种种事情，比如能力，比如开心等，从而让我们的对话有时不那么投机。
其实没什么好坏之分，每个人本就不同，至于哪种生活更好，见仁见智。

其实我们都是多虑了，大家应该都比看上去的样子要懂的更多，想的更远。

第四更：闻窗外事，读圣贤书

2019.10.20

前几天无意间读到了《蜗居》的影评,触目感怀。

用富养的方式，将高贵的性格和忠诚的价值观，给予下一代。
不只是所拥有钱的数量，而是要对于子女成长投入更多的关注。

其实，刺激我写下文字的原因，是影评的后半部分。

当虚伪的宋思明用作为公仆为人民服务的的特权来引诱年轻女孩寻欢作乐时，面目可憎。金钱和权利的影响，在剧中无孔不入，无法回避。最完美的角色，反而是美国人马克，绅士、宽容、无私的帮助女性朋友，没有企图。

月前的新加坡之行，正如许多去过的人所说，那里的基础建设，商业贸易，游戏娱乐等，是不如我国一线城市的，历史文化更是远不如中国。我的第一感受也是如此。不过，新加坡作为20多年发达国家，他的生产力水平，经济运行机制，国民的文化和国际化水平，我想还是高出中国不少的，这些软实力是需要在吃饱肚子的前提下耕耘多年才能培养出的意志。诚然，我国用一代人的时间完成了其他国家几代人的建设，但是为此也付出了代价，此处不便细表。我热爱我的祖国，在享受完发展带来的福利我也愿意去承担那不可避免的代价，但是我们还有很远的路要走，除了超一线和一线，我们还有更多不那么发达的城市，国民的收入水平可能要比我想象的要低，还有很多人是得不到足够的关注。希望我们的国家，能尽早培养出大量马克这样的人。我知道这是转型过程中的痛苦，我希望它快点过去。

面对剧中的腐败，不禁想到了前些天韩国艺人崔雪莉的非自然死亡。

权力导致腐败，绝对的权力绝对地导致腐败。
把权力关进笼子的最好的方法，就是表面看起来像民主和自由的样子，
一方阶级处于绝对的优势地位，但其他各方相对弱小的势力结合起来也可以威胁到它，这时权力就不敢作恶太过分，而大部分时间去实现它应尽的义务：管理国家，建设国家。
当有某一方势力强大到成为绝对势力，就会诞生崔雪莉这样的“离奇事件”。

富贵不能淫，贫贱不能移。这里的“淫”讲的是“过度”。
欲望无度，永远不能被满足的时候，那就是变态。
富人如果不加强自身的文化素质和精神建设，就极易产生变态的欲望。
至于依靠那微乎其微的道德操守和品德修养，更是不靠谱。

达则兼济天下，穷则独善其身。
毋以善小而不为，勿以恶小而为之。
这些从小背熟的圣贤书，往往真的能指导我们过好一生。

在这不清不楚的环境下，我只想做个好人，做一个善良的人，尽我所能的去关心身边的人，以后为子女的成长投入更多关爱，减少为他人的负担，正直的走下去。

第五更：乘风破浪，勇立潮头

2019.10.27

最近，区块链又火了。

习近平在中央政治局第十八次集体学习时强调 把区块链作为核心技术自主创新重要突破口 加快推动区块链技术和产业创新发展。

意味着我国在互联网这块又要来一波开放性政策了。

作为一名应用级开发工程师（纯调API），我真应该时刻学习，掌握架构知识了，要不然即将开始“函数即服务”，我可就没饭吃了。

区块链如何解决最核心的信任问题

  1. 去中心化
  2. 基于数学和密码学
  3. 公开透明，可追溯，不可回滚

区块链演化：

1.0 数字货币（比特币）
2.0 智能合约（以太坊）
3.0 价值互联网

以程序的角度去看待这个问题，单点故障就是中心化，高可用就是去中心化。
它像是一个没有管理员，每个节点都拥有全部数据的分布式存储系统。

多地存储会出现数据一致性的问题，区块链上叫做分叉

  • 软分叉
    临时分叉，当系统更新后版本差异导致分叉，版本统一后则分叉消除
  • 硬分叉
    由某人或组织非法拉取的的永久分支，并号召大家认可自己的分支，如“以太坊”

六层模型

如今区块链在实际生活中有多大的应用场景呢？感觉也不多，就是一些去中心化的储存。
从技术的角度出发，这是一门什么有的玩的方向，听说能开发区块链底层技术的工程师全球不超过2000人。我不奢求能学会开发，只求懂得使用。
把以太坊作为区块链的底层系统，实现EVM+API，称之为DAPP，即分布式应用程序，现在多加EAAS（区块链即服务）。
至此，微服务架构已是基础中的基础，扎实微服务思想后迈向区块链。
但是如今这蛮荒期，如果有人拉你去投资区块链，那基本可以拉黑处理了，近期肯定会有不少拿热点做传销的皮包公司会出来圈一波钱走人。

总之，跟党走，有肉吃。
